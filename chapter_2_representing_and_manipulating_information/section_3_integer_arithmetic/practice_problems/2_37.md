# Practice 2.37

You are given the task of patching the vulnerability in the XDR code shown in the aside on page 136 for the case where both data types `int` and `size_t` are 32 bits. You decide to eliminate the possibility of multiplication overflowing by computing the number of bytes to allocate using data type `uint64_t`. You replace the original call to `malloc` (line 9) as follows:

```c
uint64_t asize = ele_cnt * (uint64_t) ele_size;
void *result = malloc(asize);
```

Recall that the argument to `malloc` has type `size_t`.

[A.](#a) Does your code provide any improvement over the original?
[B.](#b) How would you change the code to elliminate the vulnerability?

---

## A.
Since in our scenario, `sizeOf(int) == sizeOf(size_t) = 32 bits` we will want to make sure the product has more than 32 bits of precision.

This improves the issues with the fact that the product of `ele_cnt` and `ele_size` can overflow if we express the product in 32 bits. But, when we provide the value to `malloc` the argument will be cast to `size_t` which is back to 32-bits and thus will still have the same issue!

So this provides no improvement!

## B.
We could just put some guards in for the overflow scenarios and return `NULL` now that `asize` can be used to check for overflow! 

```c
uint64_t requested_size = ele_cnt * (uint64_t) ele_size;
size_t allocation_size = (size_t) requested_size;

// Guard - if we can't allocate the requested size exit early
if (requested_size != allocation_size)
    return NULL;

// Continue with malloc and copy...
```
